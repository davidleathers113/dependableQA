## The blocker that invalidated the April snapshot

`deriveBillingRunwaySummary` (`src/lib/app-data.ts`) filtered ledger debits to a rolling `Date.now() - 30 days` window and returned all-null when no debit fell inside it. The unit test hardcoded April-2026 ledger dates, so once the wall clock passed mid-May the rows aged out of the window and the test threw (`typeof null === "object"`), failing `vitest run` and therefore the whole gate. It was also a real product wart: any account with no debit in 30 days silently lost its runway projection.

**Fixed in Phase 0:** the function now takes an injectable `nowMs` clock and resolves each field independently (wallet balance stays factual; average spend / projected runway / next-recharge each go `null` only when genuinely unavailable; an already-below-threshold autopay account still reports an imminent recharge). Tests pin the clock and cover the stale-history case.

## Critical blockers ‚Äî Phase 1 (RESOLVED 2026-05-29)

- ‚úÖ **Stripe wallet crediting is now idempotent and transactional.** Migration `0009_stripe_event_idempotency` (applied to prod as version `20260529204115`) adds `processed_stripe_events` (PK on `stripe_event_id`, RLS deny-all/service-role-only), a `wallet_ledger_entries.stripe_event_id` reconciliation column, and the `apply_stripe_recharge_event` RPC, which applies the credit in one transaction ‚Äî `ON CONFLICT` dedup + `SELECT ‚Ä¶ FOR UPDATE` on the billing account defeat duplicate deliveries and concurrent races. The webhook calls the RPC and only audits when it returns `true`. Tests cover credit-once / duplicate no-op / RPC-error‚Üí500 / bad-signature‚Üí400.
- ‚úÖ **The unsafe Netlify `import-dispatch` function was removed.** It ran under the service-role client trusting body-supplied `organizationId`/`batchId`/`actorUserId` and had no caller; the app route `src/pages/api/imports/dispatch.ts` already dispatches with a session-derived org. The function, its test, and `IMPORT_DISPATCH_SHARED_SECRET` were retired.

## Major risks (Phase 2‚Äì4)

- **Import batch claiming is not atomic** (`src/server/import-dispatch.ts`): read-then-write with no `status` precondition on the UPDATE ‚Üí concurrent dispatches can both claim a batch and the CSV path's plain `insert` (no `onConflict` dedupe) can create duplicate `calls`.
- **AI jobs are at-least-once, not exactly-once**: 60s default lease is shorter than a transcription+analysis run; `recoverExpiredAiJobs` can re-queue a still-running job ‚Üí duplicate `call_analyses` rows. CAS claiming is probabilistic; `attempt_count` increments from a stale JS read.
- **Protected server auth uses `supabase.auth.getSession()`, not `getUser()`** ‚Äî revoked/deleted users keep access until token expiry.
- **Tenant isolation is proven only app-side** (`.eq("organization_id", ‚Ä¶)` against mocked clients); there are **no DB-level RLS tests**.

## Known DB / advisor items

- `public.ai_jobs` has RLS enabled with **no policies** (deny-all; safe only because every access is via the service-role client ‚Äî invariant should be documented and, with `processed_stripe_events`, tested).
- `public.organizations` has an overly permissive insert policy (`with check (true)`, `supabase/migrations/0004_rls.sql`).
- **25 unindexed foreign keys** lack covering indexes (exact count from the Supabase performance advisor, 2026-05-29 ‚Äî higher than the original audit's "~11" estimate, which only counted keys with no composite-index coverage). The advisor also flags `auth_rls_initplan` (RLS policies re-evaluating `auth.*` per row), several unused indexes, and multiple-permissive-policy overlaps ‚Äî all performance, all deferred to Phase 3.
- Ringba pixel uses a query-string `api_key` compared with a non-timing-safe `===` and an O(n) scan over tenant integrations.
- Scheduled Netlify functions (`ai-dispatch-scheduled`, `ringba-api-sync-scheduled`) have no endpoint auth.
