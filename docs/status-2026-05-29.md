## The blocker that invalidated the April snapshot

`deriveBillingRunwaySummary` (`src/lib/app-data.ts`) filtered ledger debits to a rolling `Date.now() - 30 days` window and returned all-null when no debit fell inside it. The unit test hardcoded April-2026 ledger dates, so once the wall clock passed mid-May the rows aged out of the window and the test threw (`typeof null === "object"`), failing `vitest run` and therefore the whole gate. It was also a real product wart: any account with no debit in 30 days silently lost its runway projection.

**Fixed in Phase 0:** the function now takes an injectable `nowMs` clock and resolves each field independently (wallet balance stays factual; average spend / projected runway / next-recharge each go `null` only when genuinely unavailable; an already-below-threshold autopay account still reports an imminent recharge). Tests pin the clock and cover the stale-history case.

## Critical blockers (still open ŌĆö Phase 1)

- **Stripe wallet crediting is not idempotent or transactional** (`netlify/functions/stripe-webhook.ts`, `checkout.session.completed`). Read-compute-insert with no dedupe key and no lock: a duplicate Stripe delivery (Stripe retries) double-credits the wallet; concurrent events lose-update the balance. No `processed_stripe_events` table, no unique event/session constraint, no transaction. **Do not process real Stripe funds until fixed.**
- **`netlify/functions/import-dispatch.ts` trusts caller-supplied `organizationId`/`batchId`/`actorUserId` under the service-role client.** Authenticated only by a shared secret that falls back to `AI_DISPATCH_SHARED_SECRET`. Violates the repo's "never trust a body-supplied organizationId" rule. The function has **no caller in the repo** (the app route `src/pages/api/imports/dispatch.ts` dispatches directly with a session-derived org) and is slated for removal.

## Major risks (Phase 2ŌĆō4)

- **Import batch claiming is not atomic** (`src/server/import-dispatch.ts`): read-then-write with no `status` precondition on the UPDATE ŌåÆ concurrent dispatches can both claim a batch and the CSV path's plain `insert` (no `onConflict` dedupe) can create duplicate `calls`.
- **AI jobs are at-least-once, not exactly-once**: 60s default lease is shorter than a transcription+analysis run; `recoverExpiredAiJobs` can re-queue a still-running job ŌåÆ duplicate `call_analyses` rows. CAS claiming is probabilistic; `attempt_count` increments from a stale JS read.
- **Protected server auth uses `supabase.auth.getSession()`, not `getUser()`** ŌĆö revoked/deleted users keep access until token expiry.
- **Tenant isolation is proven only app-side** (`.eq("organization_id", ŌĆ”)` against mocked clients); there are **no DB-level RLS tests**.

## Known DB / advisor items

- `public.ai_jobs` has RLS enabled with **no policies** (deny-all; safe only because every access is via the service-role client ŌĆö invariant should be documented and, with `processed_stripe_events`, tested).
- `public.organizations` has an overly permissive insert policy (`with check (true)`, `supabase/migrations/0004_rls.sql`).
- ~11 foreign keys lack covering indexes.
- Ringba pixel uses a query-string `api_key` compared with a non-timing-safe `===` and an O(n) scan over tenant integrations.
- Scheduled Netlify functions (`ai-dispatch-scheduled`, `ringba-api-sync-scheduled`) have no endpoint auth.
