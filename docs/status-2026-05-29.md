## The blocker that invalidated the April snapshot

`deriveBillingRunwaySummary` (`src/lib/app-data.ts`) filtered ledger debits to a rolling `Date.now() - 30 days` window and returned all-null when no debit fell inside it. The unit test hardcoded April-2026 ledger dates, so once the wall clock passed mid-May the rows aged out of the window and the test threw (`typeof null === "object"`), failing `vitest run` and therefore the whole gate. It was also a real product wart: any account with no debit in 30 days silently lost its runway projection.

**Fixed in Phase 0:** the function now takes an injectable `nowMs` clock and resolves each field independently (wallet balance stays factual; average spend / projected runway / next-recharge each go `null` only when genuinely unavailable; an already-below-threshold autopay account still reports an imminent recharge). Tests pin the clock and cover the stale-history case.

## Critical blockers — Phase 1 (RESOLVED 2026-05-29)

- ✅ **Stripe wallet crediting is now idempotent and transactional.** Migration `0009_stripe_event_idempotency` (applied to prod as version `20260529204115`) adds `processed_stripe_events` (PK on `stripe_event_id`, RLS deny-all/service-role-only), a `wallet_ledger_entries.stripe_event_id` reconciliation column, and the `apply_stripe_recharge_event` RPC, which applies the credit in one transaction — `ON CONFLICT` dedup + `SELECT … FOR UPDATE` on the billing account defeat duplicate deliveries and concurrent races. The webhook calls the RPC and only audits when it returns `true`. Tests cover credit-once / duplicate no-op / RPC-error→500 / bad-signature→400.
- ✅ **The unsafe Netlify `import-dispatch` function was removed.** It ran under the service-role client trusting body-supplied `organizationId`/`batchId`/`actorUserId` and had no caller; the app route `src/pages/api/imports/dispatch.ts` already dispatches with a session-derived org. The function, its test, and `IMPORT_DISPATCH_SHARED_SECRET` were retired.

## Major risks (Phase 2–4)

> **All resolved as of 2026-05-29** — kept here as the original audit record. Import atomicity + AI exactly-enough processing landed in Phase 2; DB-level tenant-isolation proof in Phase 3; verified server auth in Phase 4 (see the phase sections below).

- **Import batch claiming is not atomic** (`src/server/import-dispatch.ts`): read-then-write with no `status` precondition on the UPDATE → concurrent dispatches can both claim a batch and the CSV path's plain `insert` (no `onConflict` dedupe) can create duplicate `calls`.
- **AI jobs are at-least-once, not exactly-once**: 60s default lease is shorter than a transcription+analysis run; `recoverExpiredAiJobs` can re-queue a still-running job → duplicate `call_analyses` rows. CAS claiming is probabilistic; `attempt_count` increments from a stale JS read.
- **Protected server auth uses `supabase.auth.getSession()`, not `getUser()`** — revoked/deleted users keep access until token expiry.
- **Tenant isolation is proven only app-side** (`.eq("organization_id", …)` against mocked clients); there are **no DB-level RLS tests**.

## Phase 4 — Verified server auth (RESOLVED 2026-05-29)

Protected server authorization no longer trusts an unverified cookie session.

- **Single verification point:** `src/middleware.ts` now calls `supabase.auth.getUser()` on every request — which validates the JWT against the Supabase Auth server, so a revoked, deleted, or expired user is rejected (unlike `getSession()`, which only decodes the cookie). The verified user is stored in `locals.user`; `locals.session` was removed.
- **Resolvers trust the verified user:** `requireAppSession` (`src/lib/auth/require-app-session.ts`, pages) and `requireApiSession` / `resolveRequestAppSession` (`src/lib/auth/request-session.ts`, API routes) derive identity from `locals.user` and resolve/recheck org membership per request. Public pages were migrated from `locals.session` to `locals.user`; `login.astro` uses the verified user returned by `signInWithPassword` instead of a follow-up `getSession()`.
- **Tests:** `src/middleware.test.ts` (getUser validation; revoked/invalid-token and unauthenticated → `/login`; authenticated → through with `locals.user`), plus new `require-app-session.test.ts` and `request-session.test.ts` covering no-user, no-email, no-org, active-org-not-a-member, and multiple-org resolution. `ci:verify`: **40 files / 186 tests**.
- **No DB change** — application-layer only; production schema unaffected.
